Builtin Physics vs Px5

This project uses a custom physics engine (Px5) instead of the standard UnityEngine.Physics. This allows for specialized physics behaviors tailored to the game's needs, particularly for voxel interactions and Ragdoll mechanics.

1. Custom Physics Engine

While the project runs in Unity, the underlying physics simulation is handled by Px5. This means that standard Unity physics calls (like Physics.Raycast or accessing UnityEngine.Rigidbody) may not interact with the game world as expected unless you are specifically targeting Unity's physics layer (which is often converted or used only for authoring).

2. Authoring and Conversion

To make level design and character setup intuitive, we still use standard Unity components during the authoring process. You can attach:

  • UnityEngine.Rigidbody
  • UnityEngine.BoxCollider, UnityEngine.SphereCollider, etc.
  • UnityEngine.ConfigurableJoint, etc.

At runtime, these components are automatically converted to their Px5 counterparts by the PxConverter component.

  • How it works: PxConverter reads the properties from Unity's physics components and creates the corresponding Px5 objects (e.g., PxRigidBody, PxBoxCollider).
  • Best Practice: Ensure your prefabs or objects have a PxConverter component if they need to participate in the Px5 physics simulation.

3. Scripting with Px5

When writing gameplay scripts, you must use Px5 data types and API calls instead of Unity's default physics classes.

3.1 Recommended Aliases

A common pattern is to use using aliases to map Px5 types to familiar names within your script.

// Core Types
using Rigidbody = Px5.Unity.PxRigidBody;
using Collider = Px5.Unity.PxCollider;
using Physics = Px5.Unity.PxPhysics; // Important for Raycast/OverlapSphere etc.

// Joint Types
using ConfigurableJoint = Px5.Unity.PxD6Joint;
using FixedJoint = Px5.Unity.PxFixedJoint;
using HingeJoint = Px5.Unity.PxHingeJoint;
using Joint = Px5.Unity.PxJoint;

// Collision & Query Types
using RaycastHit = Px5.UnityExtensions.RaycastHit;
using Collision = Px5.UnityExtensions.Collision;
using ContactPoint = Px5.UnityExtensions.ContactPoint;

// Collider Types
using BoxCollider = Px5.Unity.PxBoxCollider;
using SphereCollider = Px5.Unity.PxSphereCollider;
using CapsuleCollider = Px5.Unity.PxCapsuleCollider;
using MeshCollider = Px5.Unity.PxMeshCollider;

3.2 Scripting Examples

Using the aliases above allows you to write code that is very similar to standard Unity physics code, but ensures you are using the correct Px5 types.

Scene Queries (Raycasts):


using Physics = Px5.Unity.PxPhysics;
using RaycastHit = Px5.UnityExtensions.RaycastHit;

void Update() {
    Ray ray = new Ray(transform.position, transform.forward);
    
    RaycastHit[] hits = Physics.RaycastAll(ray); 
    
    foreach(var hit in hits) {
        Debug.Log($"Hit: {hit.collider.name}");
    }
}

3.3. Collision Callback Registration

Important for Mod Writers: Unlike standard Unity MonoBehaviour methods (like OnCollisionEnter, OnTriggerEnter), Px5 does not automatically call magic methods on your scripts. You must manually register for these events on the PxActor component.

You should typically register in OnEnable and unregister in OnDisable.

Example Pattern:

using Collision = Px5.UnityExtensions.Collision;

public class DummyClass: Monobehavior
{
    private Px5.Unity.PxActor pxActor;

    private void OnEnable()
    {
        if (pxActor == null)
        {
            pxActor = GetComponent<Px5.Unity.PxActor>();
        }

        if (pxActor != null)
        {
            // Register Callbacks
            pxActor.RegisterOnCollisionEnter(OnCollisionEnter);
            pxActor.RegisterOnCollisionStay(OnCollisionStay);
            pxActor.RegisterOnCollisionExit(OnCollisionExit);
            
            // Trigger Registration Example (from PxActor.cs)
            // pxActor.RegisterOnTriggerEnter(OnTriggerEnter);
        }
    }

    private void OnDisable()
    {
        if (pxActor != null)
        {
            // Unregister Callbacks to prevent errors or memory leaks
            pxActor.UnregisterOnCollisionEnter(OnCollisionEnter);
            pxActor.UnregisterOnCollisionStay(OnCollisionStay);
            pxActor.UnregisterOnCollisionExit(OnCollisionExit);
        }
    }

    // Your Callback Implementation
    void OnCollisionEnter(Collision collision)
    {
        // Handle collision logic
    }
}

4. Voxel Object Collision

For voxel-based objects, collision handling depends on the level of control you need:

Automatic Handling (Simple)

For most voxel objects, you should use the VoxelObjectProxy component.

  • It automatically handles mesh generation and collision setup.
  • You do not need to manually add collision components; VoxelObjectProxy takes care of it.

Manual Control (Advanced)

If you need finer control over the collision behavior (e.g., specific layers, mass density, or destructibility settings), you should decouple the components:

  1. Use VoxelVolume for the voxel data and mesh.
  2. Add PxVoxelColliderGenerator to handle the collision generation.
  3. Remove VoxelObjectProxy.

This manual approach allows you to tweak settings on PxVoxelColliderGenerator such as:

  • massDensity
  • useMassFromActor
  • includeLayers / excludeLayers
  • ManualAttach (for attaching to specific rigidbodies)
